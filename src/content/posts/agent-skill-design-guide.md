# Agent 觉醒之路：如何为大模型编写高阶 Skill (工具调用) 指南

> “提示词工程（Prompt Engineering）决定了 AI 能聊得多漂亮，而技能工程（Skill Engineering）决定了 AI 能把活干得多漂亮。”

在深度体验了 OpenClaw 的本地流和 Claude Code 的终端接管后，我们已经达成了一个共识：**未来的 AI 不是聊天机器人，而是能操作系统的赛博员工。**

而让这名员工长出“手脚”的核心技术，就是 **Skill（技能 / 工具调用 / Function Calling）**。

如果你还在用写 Prompt 的思维去写 Skill，你的 Agent 一定会经常崩溃。为大模型设计 Skill，本质上是在设计一套**高容错的 API 契约**。以下是我在全栈架构中总结的高阶 Skill 设计指南。

---

## 🛠️ 1. 解剖学：Skill 的三位一体

一个标准的 Skill 并不是一段简单的自然语言指令，它必须包含极其严密的三个部分：**意图（Name/Description）**、**参数契约（Schema）** 和 **反馈回路（Return）**。

### 误区：把 Description 写给人类看
在定义一个工具时，比如 `getSensorData`（获取 senseNode 传感器数据），很多人会把描述写成：“这个工具用来查询数据库里的信息。”

**这是大忌。** 大模型是依靠你的 Description 来进行路由决策的。你的描述必须包含**触发边界**和**前置条件**。

**✅ 高阶写法：**
> “根据精确的 MAC 地址提取 senseNode 设备的最新波形数据。**注意：** 仅当用户明确提供了设备的 MAC 地址或别名时才调用此工具。如果用户只给了模糊的房间号，请先调用 `queryMacByRoom` 工具。”

---

## 🛡️ 2. 防御性参数设计：对抗 LLM 的“幻觉”

大模型在生成 JSON 参数时，天生带有创造性（也就是幻觉）。如果你让它传一个时间，它可能会传 `今天`、`2026/04/05`、`2026-04-05T12:00:00Z`。这种不可控的数据一旦打入你的 Spring Boot 后端，立马触发 `Exception` 崩溃。

**如何建立抗脆弱的参数护城河？**

1. **强类型与枚举（Enums）**：
   不要用 `String` 来接收状态。如果你的 moneyNote 账单只有收入和支出，必须在 Skill 的 Schema 里把参数写死为 `enum: ["INCOME", "EXPENSE"]`。
2. **强制格式约束（Regex Pattern）**：
   对于日期、IP、MAC 地址等，直接在 Skill 层面加上正则表达式约束。大模型很聪明，它看到了正则约束，生成参数时就会乖乖就范。
3. **参数默认值与容错**：
   永远不要假设 AI 会传满所有参数。在业务代码里做好判空兜底。

---

## 🔄 3. 状态机思维：闭环反馈（The REPL Loop）

这是初级 Agent 玩家和高级架构师最大的分水岭。

初学者写 Skill：大模型调用接口 -> 接口报错抛出 500 -> 大模型懵逼，开始胡言乱语。

**高阶玩家写 Skill：必须把 Error 变成人类可读的 Prompt 返还给大模型。**

如果 AI 调用 `deleteDatabase` 工具因为权限不足失败了，你的 Java 或 Python 脚本不应该直接抛出 `AccessDeniedException` 让程序中断，而是 `catch` 住这个异常，向大模型返回一段带有指导性的文本：

```json
{
  "status": "failed",
  "error": "权限不足。由于当前处于安全沙盒模式，你无权执行删除操作。请改变策略，尝试调用 'disableDatabase' 将其标为禁用，或者向人类用户（User）请求确认。"
}
```
## 💡 结语：在硅基的混沌中，捍卫工程的秩序

很多人恐惧大模型最终会取代程序员，那是因为他们依然把自己定义为“API 的搬运工”或“业务代码的纺织工”。

但当你真正开始为 Agent 设计高阶 Skill 时，你会发现我们正在做一件维度更高的事情：**我们在给硅基大脑制定“物理法则”。**

大模型的本质是概率与混沌，它负责涌现“智能（以及幻觉）”；而人类工程师的职责，是用强类型的 Schema、严密的正则约束（Regex）、以及无懈可击的异常捕获反馈流（REPL Loop），将这种混沌的智能，收束成精准、可控的**工程资产**。

不论是摒弃数据库、走向全自动 CI/CD 的 Sefirah 数字花园，还是为 OpenClaw 的智能体编写防御性 Skill，底层的技术哲学永远是一脉相承的：

**不要用自然语言去向 AI 祈求准确率，要用冰冷而严密的系统契约去统治它。**

魔法的黑盒一旦被拆解，尽头永远是极致的工程学。去给你的 Agent 立规矩吧，造物主们。
