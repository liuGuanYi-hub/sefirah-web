# OpenClaw Agent 踩坑实录：从多源错误到多源知识检索的落地

> 原创发布于 2026-03-11 | AI Agent 部署与本地知识库构建笔记

最近在折腾 **OpenClaw**（一个强大的 AI Agent 工具链），试图构建自己的个人知识库。本以为是简单的 `docker-compose up`，结果在部署过程中遇到了一系列权限、协议和检索逻辑的错误。

---

## 1. 环境之殇：Docker 权限与网络隔离

第一个坑出现在镜像启动阶段。由于 OpenClaw 需要调用本地的文件系统来构建向量索引，Docker 的挂载权限经常导致服务挂起。

### 痛点描述
在 logs 中频繁出现 `EACCES: permission denied` 或 `ECONNREFUSED`。

### 解决方案
1. **明确用户组**：在 `docker-compose.yml` 中显式指定 `user: "1000:1000"`。
2. **网络打通**：如果 Agent 需要调用宿主机的 API（比如你本地运行的 Ollama 或 Spring Boot 接口），请务必在 Docker 中映射 `host.docker.internal`。

---

## 2. 协议错配：OpenAI 兼容接口的陷阱

OpenClaw 默认支持 OpenAI 协议，但在接入国产大模型或自建 API 转发层时，经常会因为 `Base URL` 或 `Header` 的细微差别导致 404 或 401。

### 避坑指南
* **Base URL**：很多时候你需要在 URL 末尾加上 `/v1`，或者**不加** `/v1`，这取决于你使用的中转服务。
* **模型名称映射**：确保配置中的模型字段（如 `gpt-4`）在你的中转端有对应的映射。

---

## 3. 核心：知识库挂载与向量检索

这是 OpenClaw 的精髓，也是最容易报错的地方。当你在 Web UI 提交文档后，后台可能会因为 PDF 编码、分词器（Tokenizer）不匹配而导致索引失败。

### 深度反思
* **文档预处理**：不要直接扔进包含复杂嵌套表格的 PDF。先进行 Markdown 化处理，检索的准确度会提升 50% 以上。
* **Embedding 模型选择**：本地部署时，如果 GPU 显存有限，建议使用轻量级的 `bge-small-zh`。

---

## 4. 总结：Agent 的本质是“连接”

折腾 OpenClaw 的过程让我深刻意识到：**Agent 并不是魔法，它是一系列复杂工程的组合。**
* 它的上限取决于 **LLM 的推理能力**。
* 它的下限取决于 **数据的清洗质量**。
* 而它的稳定性，则取决于你对 **网络、权限和接口协议** 的掌控力。

希望这次“踩坑记录”能让后来者少走弯路。
