# moneyNote 架构复盘：全栈视角下的状态流转与数据一致性

> 原创发布于 2026-03-24 | 个人财务系统的工程化重构记录

很多开发者的第一个全栈项目都是从“记账本”开始的，`moneyNote` 也不例外。但这绝不仅仅是一个简单的 CRUD（增删改查）练习。

当账单数据随着时间推移不断膨胀，当用户需要进行“按月、按分类、按收支”的多维度交叉查询时，如何保证前端状态的丝滑与后端数据的强一致性，成为了这个项目最大的挑战。

---

## 1. 痛点：为什么抛弃原生，走向工程化？

`moneyNote` 的早期版本只是几个简单的原生 HTML 页面拼接。但随着业务逻辑的复杂化，原生 DOM 操作的弊端彻底暴露：
* **状态失控**：新增一笔账单后，需要手动编写大量 JS 代码去局部刷新列表、更新总金额、清空表单，代码迅速腐化成“屎山”。
* **扩展困难**：一旦想要加入“图表分析”或“按月筛选”功能，整个数据流的流转就会变得难以追踪。

**破局之道是引入现代前端工程化。** 我选择了 **React**，核心目的是利用其**组件化思维**和**声明式渲染**，让 UI 真正成为状态的映射（`UI = f(state)`）。

---

## 2. 前端：复杂状态的精细化管控

在前后端分离的架构下，React 承担了极重的交互逻辑。

### 多维度状态共享
在账单看板页面，顶部有“月份筛选”，侧边有“分类筛选”，中间是“账单列表”，底部是“收支总计”。这四个组件高度依赖同一份数据源。
为了避免复杂的 Props 逐层透传（Prop Drilling），我合理规划了 React Hooks（`useState`, `useEffect`, `useMemo`）的使用：
* 利用 `useMemo` 对拉取到的全量数据进行前端侧的二次派生计算，确保在用户切换筛选条件时，能实现 **0 延迟的即时响应**，极大提升了 SPA（单页应用）的丝滑感。

---

## 3. 后端：RESTful 规范与数据强一致性

如果说前端负责“面子”，那基于 **Spring Boot** 构建的后端就负责“里子”。

### 接口设计的克制与规范
为了保证前后端对接的高效，所有 API 严格遵循 RESTful 规范：
* `GET /api/bills`：获取账单列表（支持分页与多条件 Query 过滤）。
* `POST /api/bills`：记录新账单。
* `PUT /api/bills/{id}`：修正账单。

### 数据库层面的优化 (MySQL)
财务系统最忌讳的就是数据对不上。
* **精细化表结构**：合理划分了 `users`、`bills`、`categories` 等实体，确保数据范式。
* **查询优化**：针对最频繁的查询场景（如按月份、按用户 ID 查询），在 MySQL 中建立了联合索引，避免了全表扫描导致的性能瓶颈，显著降低了接口响应延迟。
* **事务管理**：在 Spring Boot 的 Service 层引入 `@Transactional` 注解，确保在涉及多表关联操作时，数据的写入具备 ACID 特性（尤其是在未来的转账业务扩展中，这一点至关重要）。

---

## 4. 总结：从“写网页”到“建系统”的蜕变

开发 `moneyNote` 的过程，也是我个人技术栈从单点走向闭环的过程。

它让我深刻理解到：**全栈开发不仅是掌握了几门语言和框架，更重要的是拥有全局视角。** 前端要懂后端的接口为什么这么设计，后端要懂前端的状态为什么这么流转。只有两者同频共振，才能构建出“优雅且有用”的系统。
