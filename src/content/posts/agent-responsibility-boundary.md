# Agent 时代的责任边界：当机器替你做决定之后

> “把决定交给机器，并不等于把责任也交给机器。”

过去的软件通常等待人输入指令，再按照确定的规则返回结果。Agent 的出现改变了这条边界：它可以理解目标、拆解任务、调用工具、修改文件，甚至在没有逐步确认的情况下连续执行一系列动作。

能力的提升令人兴奋，但真正困难的问题并不是 Agent 能做什么，而是它做错之后，谁应该负责。

## 1. 自动执行不等于自主承担

机器可以发邮件、删文件、提交代码、调整数据库配置，但它并不拥有与人类相同的责任主体资格。它没有真正的利益，也无法理解一次错误会给某个人带来怎样的长期后果。

因此，“是 Agent 自己做的”只能描述执行路径，不能完成责任归属。真正需要追问的是：谁设定了目标？谁提供了权限？谁设计了审批规则？谁在明知风险存在的情况下仍然让它继续运行？

Agent 只是把过去分散在多个人和多个系统中的决策链，压缩成了一个看似智能的接口。

## 2. 授权必须有边界

一个成熟的 Agent 系统，不能只设计“它能调用哪些工具”，还要设计“它在什么条件下不能调用”。

读取公开文档、生成草稿和运行测试，通常可以自动化；发送外部消息、修改生产数据、支付费用和删除资源，则需要更高等级的确认。权限应该按照影响范围分层，而不是简单地给 Agent 一个万能密钥。

这和软件工程中的最小权限原则相同：让系统拥有完成任务所需的最少能力。权限越大，系统越高效，也越可能在错误判断时造成无法逆转的后果。

## 3. 人类审查不能沦为装饰

很多系统表面上保留了“人工确认”按钮，实际上只是让人快速点击通过。面对 Agent 生成的长报告、复杂代码和大量操作日志，审查者可能没有足够时间重新理解整个过程。

真正有效的人机协作，应该让 Agent 提供可读的决策理由、影响范围和不确定性，而不是只展示一个“执行成功”。人需要知道它看到了什么、忽略了什么、为什么选择这个方案，以及还有哪些替代方案。

审查的目标不是让人重新手工完成全部工作，而是让关键判断仍然保持可见。

## 4. 可追溯性是责任系统的日志

如果一次错误操作无法还原，团队就无法修复系统，只能寻找一个人来承担情绪上的责备。Agent 时代尤其需要完整的审计链：目标来源、提示词版本、使用的上下文、调用过的工具、返回结果、审批记录和最终状态。

这些记录不是为了监控某一个员工，而是为了回答事故复盘中最重要的问题：错误在哪个环节进入了系统？哪个护栏没有生效？下一次应该改变权限、流程还是模型？

没有日志的自动化只是在高速制造不可解释的结果。

## 结语：把责任写进架构

Agent 可以替我们执行越来越多的步骤，却不能替我们决定什么值得承担。未来的工程师不仅要设计能力，也要设计边界；不仅要追求成功率，也要保留拒绝、暂停和回滚的能力。

真正可靠的智能系统，不是让人从流程中消失，而是让人在最关键的地方仍然能够看见、理解并负责。
